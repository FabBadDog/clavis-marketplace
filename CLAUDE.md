# clavis-marketplace

This repository is the **plugin + module marketplace** for Clavis. Clavis core (the separate
`clavis` repo) is a slim host + microkernel that owns no UI and knows nothing about agents, panels, or
configuration; **everything a running Clavis does comes from the items in this repo.** The host clones this
working copy to `~/.clavis/marketplaces/clavis-marketplace`, compiles each item from source on launch, and loads
it over an in-process message bus.

There are no prebuilt binaries committed here - every item is **pure source, compiled on demand** by the
kernel (which shells out to `dotnet build`). A change takes effect on the next launch (modules) or on a
runtime reload (plugins).

## Git workflow

Commit directly to `main`; do not create feature branches unless explicitly asked. (The core `clavis` repo
follows the same rule.)

## Repository structure

```
clavis-marketplace/
  plugins/<name>/     a PLUGIN - its own collectible load context, reloadable at runtime
  modules/<name>/     a MODULE - loaded into the Default ALC, not unloadable
  tools/              repo scripts (Generate-MessageMap.ps1, run-tests.ps1)
  docs/               CODEMAP.md (item index), MESSAGE-MAP.md (bus wiring), design/ (design language + mockups)
  MARKETPLACE.md      the short catalog/authoring summary (frontmatter overview)
  CLAUDE.md           this file
```

The **catalog is the filesystem**: every immediate child of `plugins/` or `modules/` that carries a
`PLUGIN.md` is an item, and the folder determines its kind (frontmatter carries no `kind`). There is no
`plugin.json`, no generated `marketplace.json`.

### Files inside an item

| File | Purpose |
|---|---|
| `PLUGIN.md` | **The single source of truth** for the item: YAML frontmatter (name, version, assemblyName/pluginId, essential, dependencies, produces, privateAssemblies) plus human notes below. The kernel reads only the frontmatter; the catalog, build order, and deployment all derive from it. |
| `<Name>.csproj` / `<Name>.fsproj` | One project per item. C# for plugins (faster compile, easier agent edits); F# for contracts and most shared libraries. |
| source (`*.cs` / `*.fs`) | The implementation. |
| `tests/` | Co-located unit tests (xUnit). They carry **no** `PLUGIN.md`, so they are never loaded as plugins - they are the quality gate, run on demand (`tools/run-tests.ps1`). |
| `surface.json` | The item's **public API surface** (every public type + member), used by the self-modification lifecycle to detect breaking changes and drive semver bumps. Regenerated by the surface tool; do not hand-edit. |

### `PLUGIN.md` frontmatter keys

- `name` - the folder/catalog id (kebab-case, e.g. `git-log-panel`, `session-contracts`).
- `version` - semver; bumped (often by the lifecycle) when the surface changes.
- `pluginId` (plugin) - the PascalCase plugin id used on the bus and in logs (e.g. `GitLogPanel`).
- `assemblyName` (module) - the produced assembly (e.g. `FabioSoft.Contracts.Session`).
- `essential` (plugin) - `true` brings the plugin (with its module closure) up in startup phase 1,
  before the host reveals any UI. Reserve it for what the first usable window cannot exist without.
- `dependencies` - other marketplace items this one needs, as `{ name: <item>, version: ^x.y.z }`. Drives
  build order. Do **not** list host-provided assemblies here (Nucleus, `FabioSoft.Common/.Json/.Process`,
  `FabioSoft.Marketplace.*`) - those resolve from the host's exe-dir closure automatically.
- `produces` / `privateAssemblies` (module) - the produced DLL is routed into `~/.clavis/modules` and its
  private NuGet deps into `~/.clavis/libraries`; both are resolved on demand into the Default ALC.

## Plugin or module - which kind

- **Plugin** (`plugins/`): the default. Collectible and reloadable at runtime, so it enables
  self-modification. Use for anything that subscribes to / publishes bus messages, owns UI, or
  provides a service. Mostly C#.
- **Module** (`modules/`): only when the code **cannot be unloadable**. Two
  legitimate reasons: (1) it is a **contract** - bus message types that must have one identity across every
  plugin's load context; (2) it statically registers types (WPF `DependencyProperty`, attached properties),
  which roots them and prevents unload. If neither applies, make it a plugin.

## Best practices

- **Cross-plugin message types live in a contract module** (`*-contracts`, `FabioSoft.Contracts.*`).
  A type defined inside a plugin's assembly does **not** share identity across the per-plugin load
  contexts and silently fails to dispatch. If you need a new cross-plugin message, add it to an existing
  contract group or create a new `<area>-contracts` shared item.
- **Communicate only over the bus.** Plugins never reference each other's assemblies; they exchange the
  shared contract message types via `IBus` (publish / subscribe / request-response).
- **One project per item**, named for what it is, not its role (no `Manager`/`Helper`/`Service` suffixes).
- **Co-locate tests** under the item's `tests/`; every public surface should have a test. The lifecycle
  runs them as a gate before reload/commit.
- **Reuse the shared controls.** Before hand-building UI, check `clavis-controls` (inputs, buttons, labeled
  fields, section headers, status dots, tree browser), `clavis-rendering` (markdown, docking, motion, theme
  primitives), and `fabiosoft-editor`. If a widget is missing, **add it to `clavis-controls`** rather than
  building a one-off - so every plugin shares one look.
- **Theme via host resource keys** (`SetResourceReference`), never baked brushes/fonts. The active theme
  lives in `~/.clavis/themes`; plugins must not hardcode colors.
- **Follow the design language.** UI work must match `docs/design/CLAVIS-DESIGN-LANGUAGE.md` - the canonical
  spec (content-before-chrome, no rounded corners, color-as-meaning, the three type voices, everything
  animates), with visual mockups under `docs/design/mockups/`. It moved here from the core repo, which owns
  no UI; this is the home for all Clavis design language and mockups.
- **Declare real dependencies** in `PLUGIN.md` so build order is correct; keep them minimal.
- **Panels** announce a `PanelKindRegistration` and restore via `RestorePanel`; persist per-instance state
  through the workspace contracts rather than inventing storage.

## No-gos

- **No `DependencyProperty` / attached properties / `RegisterClassHandler` in a plugin.** They root the
  plugin's types in a global static registry and make its collectible load context un-unloadable. Use
  `INotifyPropertyChanged` for bindable state and plain CLR properties for control state. The single
  tolerated DP is `MarkdownPresenter.Markdown` in `clavis-rendering` (a Default-ALC module).
- **No new cross-plugin type inside a plugin assembly** (see above - it won't dispatch).
- **No rounded rectangle corners** anywhere (`CornerRadius="0"` or omit it). Circles/ellipses are fine; a
  status dot must be an `Ellipse`, never a small `Border`.
- **No referencing another plugin's project or types directly** - only shared contract/control assemblies.
- **No `System.Text.Json` in F# items** - use `FabioSoft.Json`. (C# plugins use `System.Text.Json`.)
- **No secrets** in `PLUGIN.md`, source, or config.

## Decision guide: what extension point fits the requirement?

Building a new plugin is the heaviest option. Before doing so, work down this list - a lighter
extension point is usually the right one:

1. **Is it agent knowledge, behavior, or a workflow?** (how the agent reasons, a reusable prompt, a coding
   convention, a multi-step task) -> Use a **Claude Code agent feature**, not a Clavis plugin:
   - a **skill** (`SKILL.md`) for a reusable capability/procedure the agent invokes;
   - a **subagent** for a delegated, sandboxed task;
   - a **CLAUDE.md rule** for a standing convention;
   - a **slash command** / **hook** for a repeatable action or guardrail.
   These live with Claude Code, not here. Clavis plugins are about the **app** (UI, bus, services), not
   about steering the agent.
2. **Is it exposing a Clavis action to the agent as a tool?** (let the agent open a panel, snapshot the
   workspace, send a prompt) -> add an **MCP tool to the `agent-gateway` plugin**, don't create a new one.
3. **Is it just displaying or editing text / markdown?** -> use the existing **`markdown-panel`** (or an
   existing panel kind), or register content through it. No new plugin.
4. **Is it a new keybinding or command?** -> add it through the **keymap + command palette** (a command +
   a binding in config), not a plugin.
5. **Is it a user setting / tunable?** -> store it in the **Configuration** store (`configuration.yaml`
   section); surface it in the **`settings`** plugin if it needs UI.
6. **Is it a visual / color / font change?** -> edit the **theme** (`~/.clavis/themes/default.yaml`), not
   code.
7. **Is it a reusable widget several plugins would want?** -> add it to **`clavis-controls`** (a shared
   module), not a per-plugin control.
8. **Is it a new cross-plugin message?** -> add it to a **`*-contracts`** group (extend an existing one or
   add a new `<area>-contracts` module).
9. **Only if none of the above fit** - a genuinely new UI surface, a new long-lived service, or new bus
   behavior that coordinates other plugins -> build a **plugin** in `plugins/`. If it must statically
   register WPF types or define shared contracts, the relevant part becomes a **module** in `modules/`.

A quick heuristic: *agent-facing -> Claude Code feature or agent-gateway tool; display-only -> existing
panel; preference -> config/theme; reusable UI -> clavis-controls; new app behavior -> plugin; new
shared message or DP-heavy control -> module.*

## Authoring a new plugin (checklist)

1. Pick the lightest extension point (decision guide above). If it is a plugin:
2. Create `plugins/<name>/` (or `modules/<name>/` for a module) with a `PLUGIN.md` (frontmatter first).
3. Add the single project file and source; depend only on shared contract/control assemblies + host-provided
   libraries.
4. If it introduces cross-plugin messages, put them in a `*-contracts` group, not the plugin.
5. Add `tests/` with xUnit tests covering the public surface.
6. Update `docs/CODEMAP.md` (the item index) and regenerate `docs/MESSAGE-MAP.md`
   (`tools/Generate-MessageMap.ps1 -CoreSrc <clavis>/src`) if you added or rewired a message.
7. Build by launching Clavis (the kernel compiles on demand) or `tools/run-tests.ps1` for the gate.
