Surfacing session status & background tasks

Claude Code emits three system subtypes Clavis currently drops as "unknown" debug noise. Here is what they carry, and ideas for turning them into visible signal — pick a direction and I'll build it.

00 What the subtypes actually carry

Because status is passed through from an SDK variable, the pill maps the known words to colours and renders any unknown value with a neutral dot and its raw text (title-cased) — so a phase we've never seen still reads, never vanishes. A null status is not a phase: it carries a permissionMode change and routes to Clavis's existing mode indicator instead.

05 My recommendation

Pair 01 + 03, design for the unknown

The two signals are different in kind, so give each its natural home rather than cramming both into one widget:

  • Phase pill (01) for status — a session-level "what am I doing" belongs in the status bar. Map known phases to colours; render any unknown value with a neutral dot and its raw word, so we're future-proof against phases we can't see today.
  • Task tracker (03) for task_* — these are discrete, countable units of work with a start and an end, which a count-pill-plus-list models exactly.
  • Mode changes ride the same subtype — a status:null message with a permissionMode should route to Clavis's existing mode indicator, not the phase pill. It even catches mode switches made outside Clavis (the agent entering plan mode, an /auto).
  • Under the hood all three subtypes get recognized (mapped to real Agent* contract events via the bridge, with status treated as nullable), so they stop logging as "unknown" regardless of how much UI we surface.

Rail whisper (02) and the unified line (04) are lower-effort fallbacks if you'd prefer to touch fewer surfaces — say the word and I'll adjust.
